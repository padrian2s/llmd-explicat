🧠 KV Cache — why we don't recompute the past

The model generates token by token. At every step, attention needs the keys and values (K/V) of all the tokens that came before. Without a cache, you recompute them at every step. With a cache, you read them from memory and only compute the new token.

📦 PagedAttention — the core innovation in vLLM

You don't know in advance how long a response will be. Classic servers reserved contiguous GPU memory for the maximum length — huge waste + fragmentation. vLLM splits the KV cache into small pages allocated on demand, wherever there's room (like virtual memory in an OS).

Identical requests enter both systems. The actual response length is random (4–16 blocks).

🧭 Inference Scheduler — round-robin vs. prefix-aware

6 users (the colors) hold multi-turn conversations with a model served by 4 pods. Each pod caches at most 3 conversations. The LLM-D question: where do you send the next message?

Let it run ~20s in each mode and compare TTFT. With round-robin, conversations land all over the place → constant recompute and the cache "thrashes". With prefix-aware, each conversation sticks to the pod that remembers it. From the meetup: up to 57× faster on real workloads.

🏥 MoE + EPLB — balancing the experts

An MoE model routes each token to a few "experts". In production, the token distribution is imbalanced: some experts run hot, others sit idle. The whole system moves at the speed of the busiest GPU. EPLB replicates the hot experts and rebalances.

16 GPU slots, 8 experts (E1 = the most requested). Without EPLB every expert gets exactly 2 slots → E1's bars shoot through the ceiling (red outline = bottleneck) and everyone else waits on it. With EPLB, E1 gets more slots, the per-slot load evens out and throughput goes up. Press "Shift the workload" with EPLB on to see the online rebalancing Tyler talked about (today it costs ~0.25s in production).
