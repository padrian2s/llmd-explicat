LLM-D, explained in plain language

Used KServe a few years ago, and meanwhile the inference world has changed completely? This guide brings you up to speed: what LLM-D is, why it exists, and what all the new terms mean.

Open Source Kubernetes-native vLLM at the core IBM · Red Hat · AMD · Google · NVIDIA

⚡ TL;DR — the 30-second version

LLM-D is an open source distributed inference stack for LLMs, launched in May 2025. It takes vLLM (the engine that actually runs the model) and orchestrates it intelligently on Kubernetes: cache-aware routing, prefill/decode separation, autoscaling, and support for huge models across dozens of GPUs.

🧠

Smart scheduler

Routes requests to the pod that already has the context in cache, instead of blindly round-robining.

✂️

Separate prefill/decode

The two phases of inference run on different instances, each optimized for its own job.

📈

An autoscaler built for AI

Scales up/down based on inference metrics, including scale-to-zero for rarely used models.

Why you should care The production numbers: from 3× better latency on long prompts, up to 57× faster on certain workloads with cache-aware routing. It's not magic — it's just avoiding work that's already been done.

1Why LLM inference isn't a regular web workload

Back in the classic KServe days, a request was a request: it came in, got processed, went out. Stateless, predictable, simple load balancer. LLM inference broke all of those assumptions.

📏

Unpredictable requests

A prompt can be 50 tokens or 50,000. Context length dramatically changes the compute and memory cost. You can't do capacity planning like you would for a REST API.

💾

State everywhere

Every conversation builds up a KV cache. If the next request lands on a different pod, everything gets recomputed from scratch. A classic load balancer has no clue about any of this.

⚖️

Expensive SLOs

Customers pay for guaranteed latency. One bad scheduling decision = wasted GPUs or a missed SLO. Both cost you serious money.

🧩

Two different phases

Inference is no longer a monolith: prefill (processing the prompt) and decode (generating the response) have completely different hardware needs.

Analogy
A classic load balancer is like a taxi dispatcher who sends customers to random cars. LLM-D is the dispatcher who knows which driver drove this customer yesterday and remembers the address — the trip gets 10 times shorter.

2KV Cache — the central concept of the whole story

If you take away one thing from this page, make it this. Almost every optimization in LLM-D revolves around the KV cache.

What is it?

Transformers understand context through self-attention: every token compares itself against all the tokens before it. That scales quadratically with context length — it gets very expensive very fast. The fix: save the intermediate results (the keys K and values V) and reuse them instead of recomputing them at every step. That's the KV cache.

Without cache vs. with cache
No cacherecompute the ENTIRE context for every generated token 🐌
With KV cachecompute only the new token, the rest comes from memory ⚡

Why does it matter so much?

Reusing a prefix from cache is at least an order of magnitude cheaper than recomputing it (prefill). Manus (an AI agent platform, now at Meta) said it outright: "KV cache hit rate is the single most important metric in production". That's why OpenAI and others charge cached tokens at 10× cheaper rates.

A concrete example from the meetup A 10,000-token prompt on Qwen 32B: the first request takes 4.3 seconds. The second identical request (cache hit): ~0.6 seconds. 7 times faster, zero wasted compute.

Where does this show up in practice?

  • Multi-turn chat — on every turn you reuse the entire conversation history so far.
  • AI agents — on every tool call, the agent's context is its complete history. The input/output ratio can be 100:1 — almost all of it is reusable prefix.
  • Parallel branching — multiple agents forking off the same shared context (tree-of-thought, swarms).
The problem The KV cache grows enormously. Even for a small 7B model, a typical conversational workload quickly reaches terabytes of KV cache. It doesn't fit in GPU memory. Hence the need for smart routing (section 3) and offloading (section 6).

3Inference Scheduler — smart routing, not round-robin

When you run multiple replicas of a model, the KV cache gets fragmented across pods. The key question: where do you send the next request?

Naive load balancer vs. cache-aware scheduler
Requestturn 2 of the conversation
→
Naive LB → Pod Bcache miss, recomputes everything 😩
LLM-D → Pod Acache hit, the context is already there 🎯

LLM-D builds on the Gateway API Inference Extension from Kubernetes (so it doesn't reinvent the wheel) and adds inference-specific routing plugins:

  • Prefix-aware routing — each pod emits events about what it holds in cache; the scheduler keeps a global map and sends the request to wherever the prefix lives.
  • Load-aware routing — looks at vLLM metrics (queue depth, utilization) via Prometheus and avoids overloaded pods.
  • The combination of both — balances cache affinity against load distribution. Extensible: you can write your own plugin (Endpoint Picker).
Benchmark result On a 16× H100 cluster: naive routing and load-aware-only routing both blow up in TTFT as traffic grows. Precise prefix-aware routing hits the cluster's theoretical maximum. On some workloads: up to 57× faster.

Analogy
It's like a restaurant kitchen: prefill is the mise-en-place (chopping vegetables intensely, over continuous heat — heavy lifting), decode is the plating (assembling plate by plate — rhythm work). If the same cook does both, they block each other. Separate them → each station gets sized for its own job.

The details that matter

A year ago this was an experimental topic; today it runs in production (Mistral and others). But it's not free: you have to orchestrate how many prefillers vs. how many decoders you run, what parallelism level each one uses, and how the KV cache gets transferred between them.

An important nuance from the meetup P/D disaggregation doesn't always help. At very low or very high concurrency, the benefit vanishes. In the middle zone you can gain ~1.5× throughput. It depends on the model and the workload — which is exactly why LLM-D offers "well-lit paths": tested recipes, so you don't have to explore the entire knob space on your own.

For KV transfer between prefillers and decoders, LLM-D uses NIXL (a transport library from the NVIDIA Dynamo stack) — peer-to-peer over RDMA, or, a recent contribution, over plain TCP (for those without expensive RDMA networks).

5MoE and Wide Expert Parallelism — today's giant models

Almost all recent large models (DeepSeek, GPT-OSS, etc.) are Mixture-of-Experts. That fundamentally changes how you serve them.

What is MoE, in short?

Instead of one huge dense layer that every token passes through, the model has hundreds of small "experts" (e.g. DeepSeek: 256 experts) and a router that picks just a few of them (e.g. 8) for each token. The model is huge on paper but activates very little per token — sparsity. Empirically, sparse models simply perform better at the same compute cost.

Analogy
A hospital: you don't send every patient to all 256 doctors. The front desk (the router) sends them to the 8 relevant specialists. Efficient — but if everyone shows up with the same problem, the cardiologist gets swamped while the dermatologist sits idle. That's expert imbalance.

The consequences for serving

↔️

From TP to EP/DP

Tensor parallelism stops making sense (the inner dimensions get too small). You move to expert parallelism + data parallelism, across multiple nodes — 16, 32, even 96 GPUs acting as a single system.

⚖️

EPLB

Expert Parallel Load Balancing: "hot" experts get replicated and rebalanced online, during inference, so no GPU becomes the bottleneck.

🔀

DBO

Dual Batch Overlap: the forward pass is split into two micro-batches, so one's compute overlaps with the other's communication. On H200: up to a 2× gain.

👥

LeaderWorkerSet

A new Kubernetes API that lets a group of pods run as a coordinated "gang" (gang scheduling), with a leader that loads the model — essential for multi-node deployments.

The rule of thumb from the meetup: decode goes wide (EP32–EP96, wants high concurrency and lots of KV cache), prefill stays narrow (EP16) — if you need more prefill throughput, you simply add more prefillers.

6KV Cache Offloading — when GPU memory runs out

GPU memory is the most expensive resource in the cluster, and the KV cache grows into terabytes. The solution: a hierarchy of storage tiers.

1. GPU memory (HBM)fastest, most expensive, smallest — this is where vLLM works natively
⚡⚡⚡
2. RAM / CPU (DRAM)asynchronous GPU→CPU offloading via DMA; comes out-of-the-box in vLLM
⚡⚡
3. Storage / Disk (POSIX)LLM-D's filesystem connector — any POSIX storage, including parallel filesystems; huge, but slower
⚡

How it works in practice: the GPU fills up → evictions begin → performance would drop, but the CPU tier picks up the cache hits → when that fills up too, the storage tier carries the workload much further. Result: the same hardware serves many more users before you "hit the wall".

Beyond vertical ("north-south") offloading, there is also horizontal ("east-west") sharing: a shared storage tier from which any pod can read KV cache computed by another — compute once, reuse anywhere. This is where LMCache or KVBM come into play as backends.

What's next (LLM-D 0.6+) Workload-aware scheduling: a scheduler that understands an agent's program (not isolated requests), context-based eviction instead of naive LRU (keep the system prompt, know that the agent off on a tool call will come back), and proactively moving state before the request even arrives. Plus support for reinforcement learning, where inference becomes the bottleneck of training.

7Coming from the KServe world? Here's what changed

KServe remains excellent for serving "classic" models (sklearn, XGBoost, small deep learning models). But LLMs have requirements that the stateless request/response architecture just doesn't cover.

The good news for you LLM-D's philosophy is "don't reinvent the wheel": everything is built on standard Kubernetes — Gateway API, CRDs, Helm charts, Prometheus. If you know how to operate a Kubernetes cluster (and you do, from the KServe era), the barrier to entry is low. On top of that, "well-lit paths" = blueprints, tested and benchmarked, for the common patterns, with a detailed blog post for each one.
The reality on the ground (AMD's lesson from the meetup) Deployments don't always work "on the first try": mismatched CRD versions, node affinity on shared clusters, RDMA/NIXL paths that don't line up with the container image. Nothing insurmountable — but verify the alignment between the vLLM image, the network drivers, and the hardware stack (ROCm on AMD, CUDA on NVIDIA) before blaming LLM-D.

9Quick glossary — all the terms in one place

Click any term for an explanation.

KV Cache

The model's working memory: the keys (K) and values (V) computed by self-attention for the tokens already processed, saved so they don't get recomputed. Reusing it is ≥10× cheaper than recomputing.

Prefill

The first phase of inference: processing the entire prompt at once. Compute-intensive. Determines time-to-first-token (TTFT).

Decode

The second phase: generating the response, token by token (autoregressive). Limited by memory bandwidth, not compute.

P/D Disaggregation

Separating prefill and decode onto dedicated instances, each sized and configured for its own phase, with KV cache transfer between them.

TTFT / Time To First Token

How long the user waits until the first word of the response appears. The number 1 UX metric, dominated by prefill (or avoided entirely via a cache hit).

MoE / Mixture of Experts

An architecture where the model has hundreds of sub-networks ("experts") and a router that activates only a few per token. A huge model on paper, with small compute per token.

EP / Expert Parallelism and Wide EP

Distributing a MoE model's experts across multiple GPUs/nodes. "Wide" = a very large number of GPUs (32–96) working as a single inference system.

EPLB / Expert Parallel Load Balancing

Online replication and rebalancing of "hot" experts, so no GPU becomes a bottleneck when the token distribution differs from the one seen during training.

DBO / Dual Batch Overlap

Splitting the forward pass into two micro-batches to overlap one's compute with the other's communication. ~2× gain on H200; pointless on GB200 (the NVLink is too fast for it to matter).

NIXL

A transport library (from NVIDIA Dynamo) for moving KV cache between nodes: peer-to-peer over RDMA or, more recently, over TCP. AMD has its own equivalent, RIXL.

LeaderWorkerSet (LWS)

A new Kubernetes API that runs a group of pods as one coordinated unit (one leader + workers), needed for serving models that don't fit on a single node.

Gateway API Inference Extension

A Kubernetes extension that allows treating inference requests with inference-specific metadata and metrics, the foundation LLM-D builds its scheduler on. The routing plugins are called Endpoint Pickers.

Prefix-aware routing

Routing a request to the pod that already has the prefix (the conversation history) in its KV cache, instead of blindly spreading the traffic.

Well-lit paths

The LLM-D philosophy: validated, benchmarked blueprints for the common deployment patterns, so you don't explore hundreds of knob combinations on your own. Each one comes with a guide + blog post.

Variant Autoscaler

LLM-D's autoscaler: reactive mode (on utilization) or proactive (deciding even the P/D configuration), with scale-to-zero for rarely used models.

SLO / Service Level Objective

The performance guarantee (e.g. latency) you promise your customers. In inference, the SLO is the currency: you bill on it, you size your cluster on it.

TP / DP — Tensor / Data Parallelism

TP: split a model's layers across multiple GPUs (the classic approach, for dense models). DP: run parallel copies that handle different requests. On MoE with MLA (DeepSeek), DP instead of TP gives you ~8× more KV cache room, because TP would replicate it.

vLLM

The open source inference engine at the core of LLM-D. It pioneered PagedAttention (efficient GPU memory management for the KV cache) and automatic prefix caching.

SGLang

An alternative inference engine to vLLM; LLM-D is adding basic support for it, so the stack becomes agnostic at the engine level too.
